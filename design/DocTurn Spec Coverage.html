The biggest gap

Agent-by-agent: the verification core is the missing piece

The spec frames a stateful hierarchical multi-agent system as DocTurn's reason for being — “nothing critical is missed.” Here is how each agent fares in v1.

Missing
Orchestrator Agent
Central brain — request typing, sequencing, final synthesis & preview. No equivalent in v1.
Partial
Data Extraction Agent
Reduced to a deterministic mock extractor. No Epic pull, no normalization/flagging of stale data.

No draft order or H&P generation; no service/urgency suggestions.

Missing-data / contradiction / completeness checks with loop-back. Entirely absent.

⚠️Headline finding: the Verification → Guardrail safety loop — the spec's central “nothing gets missed” promise — has no representation in the v1 blueprint. v1 builds a correct routing & messaging backbone, but not the clinical-safety AI that defines the product.
Compare & contrast

Two documents, two scopes

They are not in conflict — they describe different layers. The vision spec is the destination; the blueprint is a runnable first leg that intentionally defers the AI core.

Full vision spec

  • AI-first: 6 cooperating LangGraph agents with verification loops
  • Voice & text natural-language intake
  • Consult and admission workflows, each with deep checks
  • Acuity-scored live queue (1–5)
  • Manual-first routing; round-robin optional & configurable
  • Live Epic / Redox / FHIR + insurance-aware suggestions
  • Text-link fallback, auto-reminders, director escalation
  • 5 roles incl. ER Director & cross-tenant Developer

v1 buildable blueprint

  • Infra-first: one TS monolith, SQLite, WS realtime, zero secrets
  • Deterministic mock extractor stands in for all AI
  • Single generic assignment state machine
  • Flat creation-ordered queue, no acuity
  • Round-robin-centric selection (census/cap), manual flag
  • All integrations stubbed → standalone by default
  • WS notify + expiry/reroute; no SMS link, no escalation
  • RBAC + tenant scope; Director + provider roles
ℹ️Verdict: The architecture is internally honest — it openly labels itself “deliberately trimmed.” The risk is only that several headline spec features (multi-agent core, acuity, consult/admission split, text fallback, insurance routing) are not yet named in its “Later” backlog, so they could read as forgotten rather than deferred.
Recommendations

Closing the gap — what to add, and when

To make the architecture faithfully include the additional functionality (even as deferred scope), three moves:

Now
Name the deferred features explicitly. Add to the blueprint's “Later” chips: multi-agent verification, acuity scoring, consult/admission split, text-link fallback, insurance routing. Turns silent omissions into a tracked backlog.
Next
Add a Milestone 9 — Agent orchestration seam. Define the Orchestrator → Extraction → Drafting → Verification → Guardrail → Routing interface now (even mock-backed) so the safety loop has a home and the API shape is stable before real models arrive.
Next
Add an acuity field + requestType (consult/admission) to the data model. Cheap to add early; expensive to retrofit once the queue and routing assume a flat, single-type world.
Later
Schedule the AI core & integrations as their own phase. Verification/Guardrail agents, live Epic/FHIR, insurance suggestions, and the SMS Accept/Deny link land after the v1 backbone is green.

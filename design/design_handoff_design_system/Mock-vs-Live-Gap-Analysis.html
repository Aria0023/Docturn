Design Mock vs. Live App — Gap Analysis & Recommendations

A screen-by-screen reconciliation of the DocTurn design mock against the running Replit build and the Requirements & Data-Model spec (v1.0). Use this to decide what the mock should adopt from the live app, what the live app should adopt from the mock, and where the data model needs to move.

Sources Live Replit build (Developer + Director portals), Requirements & Data Model v1.0 (37 domain tables) Scope Whole system — all roles & portals Intent Flag differences · propose improvements

00 How to read this

Every row carries a status chip comparing the live app to the current mock. "Live app" is the source of truth for features; the mock is the source of truth for visual language (see the DocTurn design system). Recommendations reconcile the two.

06 Data-model actions for dev

The schema items the mock's improvements depend on. Nothing here is cosmetic — each is a build blocker for a mock-only feature.

  • Off-rotation flag. Add hospitalists.inRotation boolean (default true), independent of isWorkingToday, so a provider can be on shift yet out of round-robin. Round-robin selection must exclude inRotation = false.
  • Consult services on an assignment. The mock's multi-consultant routing has no home in assignments (one hospitalist per assignment). Add an assignment_consults join (assignmentId, service/specialty, status) or model consults as typed conversations.
  • Configurable shift types over hard-coded names. Drive shift labels/hours from organizations.shiftTypeDefinitions (jsonb) + roundRobinShiftTypes; don't hard-code "Day call / Swing / Nights" in the UI.
  • Expiry state in ER list. Surface assignments.expiresAt as an "expiring soon" (amber) + "expired" (slate) state to match the status color language.
  • Apply-to-all cap. "Default Cap for All" should write both the org default (Max Patient Cap per Provider) and each hospitalists.patientCap, clamping current census.
  • Test-data hygiene. Scrub seed/test rows (Test Role 2, ewrtwert shift, Test Bug Provider) before demo/production.

07 Recommended priorities

If you act on one section, make it the Director reconciliation — it's where the mock and live app disagree most on structure.

  1. Reconcile Director IA to the live six-tab model.Move shift-type definitions + round-robin rules into a Settings tab; keep the roster as one rotation-ordered list with inline per-provider shift editing. Add search / role filter / bulk select.
    Resolves the largest structural divergence (§3).
  2. Land the data-model changes in §6.Off-rotation flag, consult join, configurable shift types, expiry state. These unblock the mock's best ideas.
    Build blockers — do before re-mocking those features.
  3. Design the missing Director tabs: Redistribute & Approvals.Both are spec-backed and central to the director's job; neither exists in the mock yet.
    Net-new design work with clear requirements (FR-4.6, FR-3.2/3.3).
  4. Adopt the mock's ER improvements into live.2-day recently-sent history, accepted-retained, per-row reassign — all ship with existing tables.
    Quick wins, no schema change.
  5. Decide Developer-portal scope.Keep it build-only and documented, or invest in a polished operator UI. Don't half-mock it.
    Scope decision, not a build task (§2).

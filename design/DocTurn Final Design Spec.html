1 System overview

What DocTurn is, who uses it

A multi-tenant, HIPAA-oriented platform that coordinates patient assignment (ER → hospitalist) and secure clinical messaging inside hospitals, on web and mobile. Marketed publicly as DoctorHeidi. Every record is isolated to one organization; the server is authoritative for routing, expiry, and roles.

1.1 — Primary roles

  • Director — hospital admin: provider management, rotation config, reassignment, org settings, incident review.
  • ER Director — manages ER physicians; superset of ER Doctor.
  • ER Doctor — patient intake, AI extraction, create assignments.
  • Hospitalist — census, accept/decline, messaging, on-shift toggle.
  • Developer — cross-tenant platform operator: org admin, impersonation, CMS, diagnostics.

1.2 — Core capabilities

  • Authn + MFA, RBAC, session lifecycle
  • Provider rotation & capacity management
  • AI-assisted patient intake (OpenAI)
  • Round-robin & manual assignment + expiry
  • Secure messaging (direct/group/emergency)
  • Realtime WebSocket presence & events
  • SMS + push notification cascade
  • Resources, scheduling, broadcasts
  • Developer console + CMS

1.3 — Document scope

Defines the target system: architecture, stack, data model, API, auth, tenancy, workflows, integrations, frontend, security, deployment, and a milestone roadmap (M0–M16). Read alongside the three companion documents in this pack — Use Cases, Function List, and System Requirements — which trace directly to the sections here.

2 Architecture

One TypeScript monolith, two clients, live services behind interfaces

A single Express process hosts the REST API and the WebSocket server, serves the React SPA, and talks to PostgreSQL through Drizzle. The Expo mobile app shares the same backend via a typed ApiClient. Every external service is an interface with a stub fallback, so dev/CI run with zero secrets and production flips to live by env.

◆

2.3a Tenant isolation first

Every storage method takes organizationId as its first argument and filters by it. The selector helpers are the highest-risk surface — they must never cross tenants.

⛨

2.3b Server-authoritative

Routing, expiry, roles, and MFA gating are computed server-side and never trusted from the client. Thin routes (Zod validate + authorize) call a central storage module; logic lives in testable services/.

⇄

2.3c Integrations behind interfaces

A factory picks live-vs-stub at startup by env. Notification cascade WS → push → SMS degrades gracefully; stub/transport errors are swallowed + logged so the workflow always continues.

2.2 — Request lifecycle

  1. Session resolution — connect-pg-simple deserializes the docturn.sid cookie into req.user (Passport).
  2. Authorization — requireAuth + requireRole(...) assert role; handlers assertSameOrg on the target row.
  3. PHI gate — requests to /api/patients and /api/assignments append a phi_access_logs row (user, org, resource, method, IP, UA).
  4. Validation — request body parsed with the shared Zod schema.
  5. Data access — handler calls tenant-scoped storage.* (Drizzle).
  6. Audit — security-relevant actions append audit_logs; suspicious events call logSecurityIncident().
  7. Realtime fan-out — state changes broadcast over WebSocket (org-scoped); content-free push, then SMS only on ack timeout, for out-of-app delivery (see §9b).

4.7 — Assignment status state machine

create (ER) →pending— accept →accepted· census ++ · patient = assigned
pending— reject →rejected— reroute → next eligible (new pending row)
pending— expires_at reached →expired— reroute → next eligible (new pending row)
accepted— director cancel →cancelled· census −−

History is preserved: reject/expire stamps resolved_at and creates a new pending row rather than mutating the old one. Census changes only on accept (++) and cancel-of-accepted (−−).

8 Patient assignment workflow

Intake → routing → acknowledgement

8.1 End-to-end

  1. Intake (ER): note → ai-intake returns { initials, roomNumber, issueSummary, specialty }.
  2. Routing: rotation.selectNext() (round-robin) or manual hospitalistId.
  3. Persist: patient (waiting) + assignment (pending, expires_at = now + timeout).
  4. Notify: WS to target → content-free push → SMS only on ack timeout (see 9b).
  5. Acknowledge: accept (census++, assigned) / decline (reroute) / no response before expiry (reroute).

8.3 Expiry & reassignment

A setInterval (~15s, capped 50/tick) finds expired pending rows, marks expired, and reroutes. Reject reroutes immediately. If none eligible even after cap relief → patient stays waiting, emit assignment.unrouted.

8.2 Round-robin selection

// org-scoped; lowest census wins eligible = providers WHERE working AND shiftType ∈ org.shiftTypes AND census < cap AND specialty matches (if required) if eligible empty: // cap relief cap += 1 for every working provider recompute eligible sort by (census ASC, rotationOrder ASC) pick = eligible[0] advance org.rotationIndex // persist cursor return pick
⚠️Highest-risk bug: a selector that drops the org filter could route a patient to another hospital. Keep every storage method org-scoped and unit-test the invariant.

9b Notification architecture & delivery cost

Push-first delivery, SMS only on escalation — content-free payloads

This is how the established clinical-communication platforms (TigerConnect, PerfectServe, OnPage) actually move messages: the default channel is an OS push notification that carries no PHI — it only wakes the app, which then fetches the encrypted message over TLS. Plain SMS / voice are escalation endpoints, fired only when a push goes unacknowledged within a timeout. Routine traffic therefore rides free APNs/FCM push and never touches carrier-billed SMS — the lever that keeps their per-message costs near zero. DocTurn adopts the same model: WebSocket → push → (timeout) → SMS.

9b.2 Push carries no PHI

The notification payload is a content-free wake-up — title like "New assignment", no patient initials, no room, no diagnosis. The app opens and fetches the encrypted message body server-side over TLS. Same rule for SMS: code / initials only plus a deep-link, never clinical detail.

⛨

9b.3 Read / ack receipts

Because push opens an app-controlled channel, DocTurn keeps Sent / Delivered / Read + an explicit acknowledge action (plain SMS can't report "Read"). The escalation clock is driven by ack state, not by send success.

⇄

9b.4 Timeouts are org-config

ack_timeout and escalation_timeout live in org_settings (server-authoritative), so a high-acuity site can shorten the SMS fallback and a calm clinic can lengthen it — no carrier cost for routine traffic either way.

9b.5 — Configurable delivery profile (per org)

Delivery is not hard-wired. Each organization's delivery mode and, if SMS is enabled, its SMS carrier are set in the Developer console and stored in org_settings.notification_profile. The channel order, timeouts, and vendor are all configuration, never code: PushService always runs; SmsService resolves to the selected carrier adapter only when the mode includes SMS.

Delivery mode

  • Push only — WS → push → re-push/fan-out → reroute. Zero carrier cost. No SMS step.
  • Push + SMS escalation default — adds the timeout-gated SMS (then optional voice) step from §9b.1.
  • Push + SMS + voice — high-acuity sites add a voice call as the final endpoint.

ack_timeout and escalation_timeout stay org-tunable in every mode.

SMS carrier (when mode includes SMS)

// org_settings.notification_profile { mode: "push" | "push_sms" | "push_sms_voice", smsCarrier: "twilio" | "sns" | "pinpoint" | "messagebird" | "vonage", ackTimeoutSec: 90, escalationTimeoutSec: 180 }

Each carrier is an adapter behind the one SmsService.send() interface; a registry maps the smsCarrier value to its adapter + per-org credentials. Adding a carrier = one adapter, no workflow changes.

9b.5a — Configured in the Developer console

To keep it simple, the notification profile is developer-only — one place owns delivery mode, carrier choice, timeouts, and secrets. Directors and other roles don't see these controls.

⚙

Developer panel owns the full profile

  • Delivery mode per org — push only / push + SMS / push + SMS + voice.
  • SMS carrier per org — Twilio · SNS · Pinpoint · MessageBird · Vonage.
  • Carrier credentials (Twilio SID, AWS keys, etc.) — server-side only, never sent to clients.
  • Timeouts — ackTimeoutSec, escalationTimeoutSec.
  • 10DLC registration status & delivery diagnostics.

CFG Configuration & personalization

User-adjustable tweaks without a release

An additive layer over the server-authoritative, config-driven core: org policy + user preferences read live at decision time, per-org feature flags, and adaptive defaults that suggest (never silently apply) changes learned from logged usage. Full design in DT-NOTE-001.

①

C1 · Runtime settings

org_settings + user_preferences + a cached config service; validated (Zod), audited, reversible. No redeploy.

②

C2 · Behavior flags

feature_flags per org — dark-ship, gradual rollout, instant rollback, safe default when unset.

③

C3 · Adaptive defaults

suggestions mined from logged usage with evidence; a human accepts → written + audited. No ML infra.

ℹ️Boundary to hold: org policy (director-only, affects all) vs user preference (self-service). Clinical-routing behavior stays org policy — never a silent per-user change.

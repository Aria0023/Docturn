1. Introduction
1.1 Purpose & scope
DocTurn (marketed as DoctorHeidi) is a multi-tenant, HIPAA-oriented platform that coordinates patient
assignment and secure communication inside hospitals. Emergency-room (ER) physicians admit patients
and hand them off to hospitalists; the platform automates routing through a configurable round-robin
rotation, delivers real-time notifications across multiple channels, tracks acknowledgement, and maintains a
HIPAA-compliant audit trail. It ships as a React/Vite web application and an Expo React Native mobile app
that share one Express backend, session model, and PostgreSQL database.
1.2 Actors / roles
Role Description
director Hospital administrator — manages providers, schedules, round-robin config, reassignment,
and organization settings.
er_director ER administrator — manages ER physicians and oversees the ER side of assignment flow.
er_doctor / er_physician Admits patients, creates assignment requests, uses AI-assisted intake, messages providers.
hospitalist Receives assignments, accepts/declines, manages census and patient cap, communicates
with peers.
developer Platform operator — cross-tenant administration of organizations, users, CMS, integrations,
and diagnostics.
1.3 Definitions
Term Meaning
Tenant / Organization An independent hospital; all data is isolated by organizationId.
Assignment A request routing a patient from an ER physician to a hospitalist; has a lifecycle and expiry.
Round-robin Rotation algorithm that selects the next eligible hospitalist by rotation order, cap, shift, and
availability.
PHI Protected Health Information — patient initials, room, issue summary, messages, and
related records.
Census / cap A hospitalist's current patient count vs. their maximum patient cap.

2. Functional Requirements
2.1 Authentication & sessions
ID Requirement
FR-1.1 The system shall authenticate users with a username and password using server-side session-based
authentication (Passport.js).
FR-1.2 Passwords shall be stored only as salted hashes; plaintext passwords shall never be persisted or logged.
FR-1.3 The system shall support optional two-factor authentication via TOTP authenticator apps and via SMS
one-time codes.
FR-1.4 The system shall issue single-use backup codes for account recovery when 2FA is enabled.
FR-1.5 Any configured second factor shall be enforced during the primary login flow, not bypassable through
alternate paths.
FR-1.6 The system shall maintain server-side session records, support session expiry/timeout, and allow
administrative termination.
FR-1.7 The system shall record last-login time and track active presence/online state per session.
2.2 Multi-tenancy & organizations
ID Requirement
FR-2.1 Every domain record shall be scoped to an organization; users shall only read or write data within their own
organization.
FR-2.2 Organizations shall be identified by a unique code (e.g. MAYO) and be independently configurable
(timezone, branding, feature flags).
FR-2.3 Each organization shall configure assignment timeout, round-robin rules, and custom shift-type definitions.
FR-2.4 Per-organization, per-portal feature toggles shall control availability of features such as SMS, AI chatbot,
and integrations.
FR-2.5 Developers may operate across tenants for administration; all such access shall be audited.
2.3 User & provider management
ID Requirement
FR-3.1 Directors shall create, edit, deactivate, and delete provider accounts within their organization.
FR-3.2 The system shall support a self-registration request workflow that requires director/approver review before
an account is created.
FR-3.3 A provider-registration approval pipeline shall capture applicant details (name, email, phone, specialty,
license, credentials) and track status (pending / approved / rejected / on_hold).
FR-3.4 Role types and their permissions/portal access shall be configurable, including custom roles per
organization.

2.4 Patient assignment workflow
ID Requirement
FR-4.1 ER physicians shall create a patient record (initials, room, issue summary) and an assignment request.
FR-4.2 The system shall select the target hospitalist via round-robin, honoring rotation order, patient cap, shift type,
on-call and availability filters as configured by the organization.
FR-4.3 ER physicians/directors may also assign a specific hospitalist manually, overriding round-robin.
FR-4.4 Each assignment shall have the lifecycle: pending → accepted / rejected / expired / cancelled, with
timestamps for each transition.
FR-4.5 Assignments shall expire automatically after the organization's configured timeout; expiry and rejection shall
trigger re-routing to the next eligible provider.
FR-4.6 Directors shall be able to cancel or reassign assignments.
FR-4.7 Accepting an assignment shall update the hospitalist's current patient count and the patient's assigned
provider.
FR-4.8 Users shall be able to chat directly on an assignment (assignment-scoped messages with read tracking).
2.5 Notifications
ID Requirement
FR-5.1 On assignment events, the system shall notify the target provider in real time via WebSocket.
FR-5.2 The system shall send SMS notifications (Twilio) and track delivery and acknowledgement, including the
provider message SID.
FR-5.3 The system shall support push notifications (Firebase Cloud Messaging) to registered iOS/Android/web
device tokens with priority and deep-link/web-fallback handling.
FR-5.4 Providers shall be able to acknowledge assignments via SMS reply, updating assignment state.
FR-5.5 The system shall support advance/expiry-warning notifications before an assignment times out.
2.6 Messaging & directory
ID Requirement
FR-6.1 The system shall provide direct, group, and emergency conversations scoped to an organization.
FR-6.2 Messages shall support text and attachments (image/file), replies (threading), editing, and soft deletion.
FR-6.3 Per-recipient delivery status (sent / delivered / read / failed) shall be tracked, with push and SMS-backup
flags.
FR-6.4 A provider directory shall show display name, title, specialty, contact, on-call status, and availability.

3. Non-Functional Requirements
3.1 Security
ID Requirement
NFR-1.1 All authorization shall be enforced server-side for every route, WebSocket action, and record, independent
of frontend route guards.
NFR-1.2 All reads/writes shall be scoped to the authenticated user's organizationId to prevent cross-tenant access.
NFR-1.3 The server shall set hardened HTTP headers (Helmet) and apply tiered rate limiting on authentication and
general API routes.
NFR-1.4 Sessions shall use HttpOnly cookies; traffic shall be served over HTTPS in production.
NFR-1.5 Secrets and integration credentials shall be held server-side and never exposed to clients, logs, or
diagnostics.
NFR-1.6 Non-production auth/test helpers shall be unreachable in production.
3.2 Compliance (HIPAA)
ID Requirement
NFR-2.1 The system shall maintain HIPAA-compliant audit trails for all PHI access and administrative actions.
NFR-2.2 Responses, logs, and admin tools shall never expose password hashes, session identifiers, backup codes,
secrets, or PHI to unauthorized users.
NFR-2.3 Message content shall be flagged for encryption and patient-related access shall capture a
reason-for-access.
3.3 Performance & availability
ID Requirement
NFR-3.1 Assignment and messaging events shall propagate to connected clients in real time via WebSocket, with
SMS/push fallback.
NFR-3.2 Public and low-auth endpoints shall avoid expensive operations triggerable by unauthenticated users and
shall be rate-limited.
NFR-3.3 Production deployment shall support horizontal auto-scaling behind a reverse proxy.
3.4 Usability & accessibility
ID Requirement
NFR-4.1 The UI shall be responsive and mobile-first, with touch-friendly controls and accessible focus states and
contrast support.
NFR-4.2 Copy shall use plain, everyday language; status shall be conveyed with both color and text/icon.

4. Data Model — Conventions
• Source of truth: shared/schema.ts (Drizzle ORM). There are 38 tables: 37 domain tables plus the
sessions store.
• PK = primary key; FK→table = foreign key reference; NOT NULL = required. Defaults are shown where
defined.
• serial ids are auto-incrementing integers; uuid ids default to a random UUID.
• text[] / integer[] are array columns; jsonb stores structured JSON.
• Timestamps default to now() where indicated; audit/PHI timestamps are server-set.
• Every tenant-scoped table carries organizationId (FK→organizations) to enforce isolation.

12.1 Key relationships
• organizations is the tenant root; nearly every table references it via organizationId.
• users 1→N hospitalists, patients (as ER doctor), and assignments.
• patients 1→N assignments; an assignment references the patient, the hospitalist, and the ER doctor.
• conversations 1→N messages 1→N message_delivery_status (unique per recipient); messages
self-reference via parentMessageId for replies.
• departments 1→N beds, equipment, resource_alerts, and resource_metrics.
• emergency_broadcasts 1→N broadcast_acknowledgments.

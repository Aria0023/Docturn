1. System Overview
DocTurn is a multi-tenant, HIPAA-oriented SaaS platform that coordinates patient assignment and secure
clinical communication inside hospitals. Emergency-room (ER) physicians admit patients and hand them
off to hospitalists; the platform automates who receives each patient through a configurable round-robin
rotation, notifies the selected provider in real time across multiple channels, tracks acknowledgement, and
provides a HIPAA-compliant messaging layer for all providers.
The product serves several distinct roles across many independent hospital organizations (tenants). Each
tenant's data — users, patients, assignments, messages, audit records, and configuration — is isolated by
an organizationId that is enforced at the data-access layer.
1.1 Primary user roles
Role Responsibilities
director Hospital administrator. Manages providers, shift definitions, round-robin
configuration, reassignment, and organization settings.
er_director ER administrator. Manages ER physicians and oversees ER-side assignment flow.
er_doctor / er_physician Admits patients, creates assignment requests, and uses AI-assisted patient intake.
hospitalist Receives patient assignments, accepts/declines, manages census and patient cap,
and communicates with peers.
developer Platform operator. Cross-tenant administration: organizations, users, CMS,
integrations, diagnostics, impersonation.
1.2 Core capabilities
• Patient assignment workflow — ER physicians create assignments that are routed to hospitalists via
round-robin or manual selection, with configurable expiration.
• Real-time notifications — WebSocket push, Firebase Cloud Messaging push, and Twilio SMS
fallback.
• Secure messaging — direct, group, and emergency-broadcast conversations with delivery/read
receipts and audit logging.
• AI assistance — OpenAI-powered extraction of structured clinical data from free text and
natural-language command processing.
• Multi-tenancy — organization-scoped data isolation with per-tenant branding, feature flags, and
integration credentials.
• HIPAA controls — comprehensive audit trail, PHI access logging, short session timeouts, MFA, and
encryption in transit.
• Cross-platform clients — a React/Vite web SPA and an Expo React Native mobile app sharing one
backend, session model, and database.

2. Architecture
DocTurn is a TypeScript monolith. A single Express server hosts the REST API and a WebSocket server,
serves the compiled React SPA, and talks to PostgreSQL through Drizzle ORM. The Expo mobile app is a
separate client that consumes the same API and session model. External services (Twilio, OpenAI,
Firebase, Amion) are reached server-side so that secrets never touch the client.

2.2 Request lifecycle
• Ingress — TLS terminates at the platform edge; traffic reaches Express on a single port that serves
both API and static assets.
• Security middleware — Helmet sets HTTP security headers; tiered express-rate-limit guards auth and
general API routes.
• Session resolution — Passport deserializes the session from the PostgreSQL-backed store using the
docturn.sid cookie.
• Authorization — route handlers assert the caller's role and that the target record matches the caller's
organizationId.
• Data access — handlers call the storage layer (IStorage), which executes Drizzle queries scoped by
tenant.
• Audit — HIPAA middleware and explicit audit inserts record security-relevant events and PHI access.

• Real-time fan-out — state changes broadcast over WebSocket to the relevant tenant/users; push and

2.3 Design principles
• Tenant isolation first — every read and write is scoped by organizationId; cross-tenant access is
a primary threat.
• Thin routes, central storage — business data access lives behind the IStorage interface to keep
route handlers focused on validation and authorization.
• Server-authoritative state — security-sensitive values (assignment routing, expiry, roles) are
computed on the server, never trusted from the client.
• Secrets stay server-side — third-party SDK calls are made by the backend; clients never receive
integration credentials.
• Graceful degradation — notifications cascade (WebSocket → push → SMS) and integrations fall back
to local behavior when unconfigured.

4. Data Model
The schema is defined in shared/schema.ts using Drizzle ORM and is shared between the server and
(via the Zod insert schemas) the clients. It comprises 37 domain tables plus the session table. The
organizations table is the tenancy root; nearly every other table carries an organizationId foreign
key.
4.1 Modeling conventions
• Each table has a serial id primary key, except conversations and messages, which use UUIDs.
• Enumerations are expressed as text({ enum: [...] }) columns (e.g. assignment status, shift
type, risk level).
• Array columns use the column-method form, e.g. text().array().
• For every insertable table, a createInsertSchema(...).omit({ id, createdAt, updatedAt
}) Zod schema and inferred insert/select types are exported.
• JSON-heavy CMS and configuration tables use jsonb columns (shift-type definitions, landing/contact
page content, portal feature flags).

5.7 WebSocket server
A WebSocket server is mounted at /ws. On connection the server parses the docturn.sid cookie,
unsigns it, and looks up the session row to resolve passport.user. Connections that fail session
resolution or have no valid organization are closed with code 1008. Each socket is stored in a clients map
keyed by user, carrying its userId and organizationId so that all fan-out stays tenant-scoped.
Message type Direction Meaning
CONNECTION_ESTABLISHED server → client Handshake acknowledgement with
connection id.
USER_PRESENCE_CHANGED server → org A user connected/disconnected
(org-scoped).
typing_start / typing_stop client → server Typing intent for a conversation.
user_typing server → participants Typing indicator relayed to conversation
members.
MESSAGE_RECEIVED server → users New message / delivery
acknowledgement.
PROVIDER_CREATED server → org A new provider was created in the
tenant.
HOSPITALIST_DELETED server → org A provider was removed from the
tenant.
PING / PONG both 20-second heartbeat that also updates
presence.
Two helpers govern delivery: broadcast(message, organizationId) fans out to every socket in a
tenant, and sendToUsers(userIds, message) delivers to specific participants (e.g. the two members
of a direct conversation).

6. Authentication & Authorization
6.1 Authentication
Authentication uses Passport with a local username/password strategy. Login is organization-aware:
credentials are validated against the user record within the resolved organization.
• Password hashing — scrypt (crypto.scrypt) with per-user salt; legacy bcrypt hashes are still
verified for backward compatibility.
• Session store — server-side sessions persisted in PostgreSQL; the cookie carries only the signed
session id.
• Cookie — docturn.sid, httpOnly; in production secure and sameSite=strict.
• Session lifetime — 15-minute max age in production (HIPAA), 24 hours in development; rolling
resets expiry on activity.
• Registration — self-registration creates a pending record; a director must approve and assign a role
before the account can authenticate.
6.2 Multi-factor authentication
MFA is handled by a dedicated two-factor service and supports both TOTP (authenticator apps via
speakeasy) and SMS one-time codes (via Twilio).
• On login with MFA enabled, the server returns 202 with twoFactorRequired: true and holds a
pending user id in the session.
• The challenge is completed at /api/2fa/complete-login, which finalizes the authenticated
session.
• Ten one-time backup codes are generated at TOTP setup and stored as an array; each may be
redeemed once if the primary device is unavailable.
6.3 Role-based access control
Authorization is enforced server-side in route handlers and mirrored in the client for UX. The web client
wraps protected routes in a ProtectedRoute component, but the server remains the source of truth.
• Roles: director, er_director, er_doctor/er_physician, hospitalist, developer.
• Each handler checks the caller's role and that the target record's organizationId matches the
caller's (developers may act cross-tenant).
• A HIPAA compliance service centralizes permission semantics such as view_all,
create_assignments, and view_assigned.
• Configurable role_types allow per-tenant permission and portal-access definitions stored as JSON.

7. Multi-Tenancy
Tenancy is the platform's central design constraint. Every clinically or administratively significant table is
scoped by organizationId, and the storage layer exposes organization-aware query methods so that
handlers cannot accidentally read or write across tenants.
• Isolation — storage methods such as getWorkingHospitalistsByOrganization and
getActivePatientsForOrganization always filter by tenant.
• Per-tenant configuration — round-robin shift types and timeouts, branding/CMS, and feature flags live
in organizations, portal_configurations, and CMS tables.
• Per-tenant integration credentials — Twilio, Firebase, and Amion credentials are stored per
organization rather than globally.
• Real-time scoping — WebSocket broadcasts are keyed by organizationId; presence and status
never leak across tenants.
• Cross-tenant operations — reserved for the developer role through explicit /api/dev/** endpoints.
The round-robin selector is called out in the threat model as especially sensitive: a selector that ignored
tenant scope could silently route a patient to a provider in another hospital. Rotation helpers therefore
operate strictly within a single organization.

8. Patient Assignment Workflow
The assignment workflow is the operational heart of DocTurn. It connects ER intake to hospitalist
acceptance through automated routing and multi-channel notification.
8.1 End-to-end flow
• Intake — an ER physician submits a patient (optionally pasting free-text notes that OpenAI converts
into structured fields and a suggested specialty).
• Routing — the server selects a hospitalist via round-robin or accepts a manual selection.
• Persistence — a patients row and an assignments row (status pending, with expiresAt) are
created.
• Notification — the provider is notified by WebSocket, Firebase push, and/or Twilio SMS.
• Acknowledgement — the provider accepts or declines; acceptance moves the status to accepted and
updates census, decline/expiry triggers re-routing.
8.2 Round-robin rotation
Rotation logic lives in the storage layer (getNextHospitalistInRotationByOrganization) and the
routes layer.
• Eligibility — candidates must be active today, match the required shift type, and be below their
patientCap.
• Selection — the standard path favors the lowest current patient count; a rotation-index path (persisted
per organization in system_settings) cycles providers sequentially.
• Cap relief — if every eligible provider is at capacity, each provider's patientCap is incremented by
one so the queue can still drain.
8.3 Expiration & reassignment
When an assignment is created the server computes expiresAt from
getAssignmentTimeout(organizationId). If the provider does not respond before that time, the
assignment is treated as expired and becomes eligible for automatic or director-initiated reassignment to
the next provider in rotation.

9. External Integrations
All third-party calls are made server-side. Credentials are either environment secrets or, for tenant-specific

• Secrets — AI_INTEGRATIONS_OPENAI_API_KEY and AI_INTEGRATIONS_OPENAI_BASE_URL

11. Security & HIPAA Compliance
11.1 Transport & HTTP hardening
• TLS terminates at the platform edge; production cookies are secure and sameSite=strict.
• Helmet sets standard security headers (HSTS, X-Content-Type-Options, X-Frame-Options,
Referrer-Policy). The CSP is relaxed in development for Vite and should be tightened in production.
• Tiered rate limiting: stricter limits on auth routes (≈100 / 15 min) and a broader API limit (≈1000 / 15
min).
11.2 Audit & PHI logging
• Audit logs — authentication events, user management, assignment operations, CMS edits, and
conversation deletions are recorded with actor, resource, details, IP, user agent, and risk level.
• PHI access logs — a dedicated trail records which patient and which data fields were accessed and
how.
• Middleware — HIPAA audit and PHI-access middleware automatically log activity on sensitive
endpoints; response-body logging in the server entrypoint is treated as security-sensitive because it
can capture API payloads.
11.3 Session & account controls
• Short (15-minute) production session timeout with rolling renewal and server-side inactivity
enforcement.
• MFA (TOTP + SMS) with one-time backup codes.
• Director approval gate for new account activation.
11.4 Threat-model focus areas
• Broken access control across the very large route surface — every route must assert role and tenant.
• Cross-tenant leakage from a missing organizationId filter — especially in rotation/selection
helpers.
• Production reachability of dev/test helpers — quick-login, impersonation, and self-test endpoints
must be unavailable to production users.
• Information disclosure — responses, logs, and diagnostics must never expose password hashes,
session ids, backup codes, secrets, or PHI.
See threat_model.md in the repository for the full asset inventory, trust boundaries, and STRIDE breakdown.

System shape

One process, two clients, everything stubbed

A single Express server hosts the REST API and a WebSocket server, serves the React SPA, and talks to SQLite through Drizzle. External services sit behind interfaces with local fallbacks, so the whole thing boots with no API keys.

Data model

Ten core entities, all scoped by organization

Drizzle schema in shared/schema.ts is the single source of truth. Nearly every table carries an organization_id that every query filters by.

organizations

Tenant root — name, code, timeout, rotation cursor.

users

All accounts; role + hashed password.

hospitalists

Provider profile: census, cap, working, rotation order.

patients

PHI-bearing — initials only, room, complaint, status.

assignments

The routing lifecycle: status, via, expires_at.

conversations

direct · group · emergency; participant ids.

messages

Content + sender + timestamp.

message_delivery_status

Per-recipient delivered / read.

audit_logs

Lightweight HIPAA trail: action, resource, risk.

sessions

Managed by the session store (not hand-modeled).

Core workflow

The assignment state machine

The operational heart: ER intake → round-robin routing → provider acknowledgement. Server-driven, fully org-scoped, and the focus of the test suite.

create → pending — accept → accepted · census ++
pending — reject → rejected — reroute → next eligible provider (new pending)
pending — expires_at → expired — reroute → next eligible provider (new pending)
accepted — director cancel → cancelled · census −−

Round-robin selection

// org-scoped; lowest census wins eligible = providers WHERE working AND shiftType ∈ org.shiftTypes AND census < cap AND specialty matches if eligible empty: // cap relief cap += 1 for all working recompute eligible sort by (census ASC, rotationOrder ASC) pick = eligible[0]; advance org.rotationIndex

Invariants the tests assert

  • Census changes only on accept (++) and cancel-of-accepted (−−).
  • Reject/expire always yields exactly one new pending — or zero if none eligible.
  • Selection never crosses tenants and never exceeds cap (unless cap relief raised all).
  • Lowest-census provider is preferred; ties broken by rotation order.
  • Messages never reach a non-participant.
⚠️Highest-risk bug: a rotation selector that forgets the org filter could route a patient to another hospital. Keep every storage method org-scoped.
API surface

REST + WebSocket, grouped by domain

Cookie-session auth; every privileged route asserts role + tenant. Full request/response shapes live in API.md.

Representative REST endpoints

  • POST /api/login · GET /api/user · POST /api/logout
  • GET /api/hospitalists · /working · /directory
  • PATCH /api/hospitalists/:id/working-status
  • POST /api/patients/extract → mock AI fields
  • POST /api/assignments round_robin · manual
  • PATCH /api/assignments/:id/accept · reject · reassign · cancel
  • GET/POST /api/messaging/conversations · /send · /mark-read
  • GET/PATCH /api/org/config · POST /api/round-robin/reset

WebSocket events /ws

  • CONNECTION_ESTABLISHED · PING/PONG (20s)
  • ASSIGNMENT_CREATED → target provider
  • ASSIGNMENT_UPDATED → org (status change)
  • MESSAGE_RECEIVED → participants
  • typing_start/stop → user_typing
  • USER_PRESENCE_CHANGED → org
  • PROVIDER_CREATED / UPDATED → org

Cookie→session auth on connect (close 1008 on failure); all fan-out tenant-scoped via broadcast / sendToUsers.

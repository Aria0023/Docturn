0 Method & a note on tooling

How this was produced

No dedicated "systems-engineer" tool was available, so this is a hand-traced audit applying standard requirements-engineering practice. For each requirement the trace column (UC · F) was inverted to build two coverage maps: UC → requirements (is every behavior specified & verifiable?) and Function → requirements (is every function exercised by a shall-statement?). Findings are then classified by severity and given a recommended action.

Severity reflects priority × exposure: a Must-priority or security/PHI-touching gap is High/Med; a Could-priority expansion feature with no requirement yet is Low (expected — its requirements are authored at its milestone).

Coverage verdicts

CoveredPartialOpen / Gap

Covered = ≥1 dedicated requirement. Partial = covered only indirectly/transitively. Open = no requirement yet.

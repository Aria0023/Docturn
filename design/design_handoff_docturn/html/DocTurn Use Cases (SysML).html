3 Use-case specifications

Detailed flows — the operational core

Full specs for the highest-value use cases. Remaining use cases follow the same template; the summary index at the end lists all 28 with their actors and trace.

UC-01AuthenticateMust
Actors
Any human actor (primary), Twilio (secondary, for SMS OTP path)
Precondition
Account exists and is approved within an organization.
Main flow
  1. Actor submits org code, username, password.
  2. System validates credentials (scrypt/bcrypt) scoped to the org.
  3. If twoFactorEnabled → respond 202 { twoFactorRequired } and set pendingMfa (→ UC-02).
  4. Else establish session, return sanitized user, route to role dashboard.
Alt / exception
  • Bad credentials → 401, inline error, append audit_logs (auth.login.fail).
  • Unknown org code → generic 401 (no enumeration).
Postcondition
Authenticated session with 15-min rolling lifetime, or MFA pending.
Traces to
FR-AUTH-01/02, SEC-01
UC-07Create assignmentMust
Actors
ER Doctor / ER Director (primary)
Precondition
Patient record exists (waiting); caller authenticated, same org.
Main flow
  1. Caller chooses mode: round_robin or manual.
  2. Round-robin → rotation.selectNext() picks lowest-census eligible provider (cap relief if none); manual → caller supplies hospitalistId.
  3. System inserts assignment (pending, expires_at = now + org.timeout).
  4. Notification cascade fires: WS → push → SMS (→ UC-15).
Alt / exception
  • No eligible provider after cap relief → patient stays waiting, emit assignment.unrouted.
  • Manual target over cap → allowed (explicit override), audited.
Postcondition
Exactly one pending assignment targeting one provider; org rotation cursor advanced.
Traces to
FR-ASG-01/02/03, PR-02, SEC-07
UC-08Accept / decline assignmentMust
Actors
Hospitalist (primary, own assignment only)
Precondition
A pending assignment targets the caller.
Main flow
  1. Accept → status accepted, set patient.assigned_hospitalist_id, patient assigned, census++, stamp resolved_at.
  2. Decline → status rejected, stamp resolved_at, immediately reroute to next eligible (→ UC-09 logic).
  3. Emit ASSIGNMENT_UPDATED org-scoped.
Alt / exception
  • Assignment already resolved → 409.
  • Caller is not the target → 403.
Postcondition
Census changes only on accept; a decline yields exactly one new pending row when a provider is eligible.
Traces to
FR-ASG-04/05, RR-01
UC-09Auto-expire & rerouteMust
Actors
System (time-triggered) — no human initiator
Precondition
One or more pending assignments with expires_at ≤ now.
Main flow
  1. Interval (~15s) lists expired pending rows, capped at 50 per tick.
  2. Each is marked expired (resolved_at stamped).
  3. rotation.selectNext() creates a new pending row for the next eligible provider; notify.
Alt / exception
  • No eligible provider → patient remains waiting, assignment.unrouted emitted; UI shows "awaiting capacity".
Postcondition
No expired-pending row remains unresolved; history preserved.
Traces to
FR-ASG-06, RR-02, PR-03
UC-11Send secure messageMust
Actors
Any authenticated participant (primary)
Precondition
Caller is a participant of the conversation (direct/group/emergency).
Main flow
  1. Caller posts content to a conversation.
  2. System inserts messages row + a message_delivery_status row per participant (delivered = now for connected users).
  3. Emit MESSAGE_RECEIVED to participants only; recipients mark-read turns ticks blue.
Alt / exception
  • Non-participant attempts read/send → 403 (never delivered).
  • Delete → soft-delete + audit_logs row.
Postcondition
Message persisted with per-recipient delivery/read state; no leak to non-participants.
Traces to
FR-MSG-01/02/03, SEC-04
UC-22Impersonate (audited)Should
Actors
Developer (primary, cross-tenant)
Precondition
Caller holds developer role.
Main flow
  1. Developer selects an org + target user.
  2. System writes a full audit entry (phi_access_logs + audit_logs) before swapping session context.
  3. Session now scoped to the target org/user; banner indicates impersonation.
Alt / exception
  • Non-developer → 403; suspicious attempt → logSecurityIncident().
Postcondition
Impersonation session active and fully attributable in the audit trail.
Traces to
FR-DEV-02, SEC-03, SEC-06
UC-29Adjust settings (tweak)Should
Actors
Director (org policy); any user (own preferences)
Precondition
Authenticated; the setting key exists in the typed config schema.
Main flow
  1. Actor opens Settings and changes a tweak (e.g. assignment timeout, default routing mode).
  2. System validates the value against its type + bounded range (Zod).
  3. Persist to org_settings (policy) or user_preferences (preference); append audit_logs (old→new).
  4. Config cache invalidated; the next decision in the relevant service reads the new value — no redeploy.
Alt / exception
  • Non-director edits org policy → 403.
  • Out-of-range value → 400, current value retained.
Postcondition
Behavior reflects the tweak on the next action; the change is attributable and reversible.
Traces to
FR-CFG-01/02/03, F17.1 / F17.3

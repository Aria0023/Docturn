Design note · configurability & adaptive behavior

Let users make the little tweaks — no new release required.

A design note on making DocTurn self-adjustable at runtime: hospitals and providers change behavior through in-app settings (instant effect, no deploy), and the system learns usage patterns and suggests better defaults. The important part — this is a natural extension of the existing server-authoritative, config-driven architecture, not a new dependency or a pivot.

1 Why this fits the current design

The pattern is already here

DocTurn already computes behavior on the server from values stored in the database and read live at decision time. "User-adjustable tweaks" is mostly generalizing that — not inventing it.

Already runtime-configurable today

  • organizations.assignment_timeout_min — the expiry window, read by the expiry service every tick.
  • organizations.round_robin_shift_types — which shifts rotate, read by rotation.selectNext().
  • organizations.rotation_index — the fairness cursor, persisted and advanced live.
  • hospitalists.patient_cap / working — capacity & eligibility, editable in-app.

Change any of these through the Director dashboard and the next assignment behaves differently — with no redeploy. That's the seed of the whole idea.

What's missing to make it general

  • A typed settings store beyond a handful of org columns (so new tweaks don't need a schema change each time).
  • User-level preferences, not just org-level policy.
  • Behavior flags to roll a change out per-hospital without a release.
  • An adaptive layer that turns logged usage into suggested defaults.
✓All four are additive — a few small tables and a config-reading service. No existing table or flow is rewritten.
2 Three tiers, easiest first

From in-app settings to a system that learns

Adopt in order. Each tier delivers value on its own; you do not need the ambitious one to ship the cheap one.

①

Runtime settings

In-app preferences, read at decision time, instant effect.

  • Org policy (director) + user prefs (everyone)
  • Key/value store, typed & validated (Zod)
  • Edited in Settings UI → next action uses it
  • Effort: small · Risk: low
②

Behavior flags

Ship a behavior dark; flip it per hospital, no release.

  • Per-org boolean / variant flags
  • Gradual rollout & instant rollback
  • A/B a new routing rule on one site
  • Effort: small · Risk: low
③

Adaptive defaults

Learn from usage; suggest a better default a human confirms.

  • Heuristics over data you already log
  • Surfaced as a suggestion, never auto-applied
  • Starts with no ML infrastructure
  • Effort: medium · Risk: medium

4 How it's built

A thin configuration layer + an analytics-driven suggestion loop

Data model additions

org_settingskey · value(json) · type · updated_by · updated_at — org policy tweaks
user_preferencesuser_id · key · value(json) — per-person preferences
feature_flagsorg_id · flag · enabled / variant — dark-ship & rollout
suggestionsscope · key · proposed_value · evidence · status (pending/accepted/dismissed)

Values are typed & validated with Zod, exactly like the rest of the schema. A small config service loads + caches them so reads in hot paths (rotation, expiry) stay fast.

The suggestion loop (Tier 3)

// runs periodically, org-scoped, read-only analyze(orgId): // mine data you ALREADY log m = median(acceptTime from assignments) if m < org.timeout * 0.6: propose('assignment_timeout', ceil(m), evidence="median accept 4m vs 10m window") byKeyword = topSpecialty(per complaint, assignments) for kw, spec in byKeyword: propose('specialty_map', {kw: spec}, evidence=…) // nothing changes until a human accepts: accept(suggestion) → write org_settings + audit_logs dismiss(suggestion) → record, stop re-proposing

No ML, no new infrastructure — it's aggregate queries over assignments, audit_logs, and messages, surfaced as a review card.

5 Design principles for a self-adjusting system

Keep it safe, legible, and reversible

Suggest, don't impose

The system proposes; a human applies. Especially for anything that affects clinical routing or safety — never a silent change.

Everything audited & reversible

Every applied tweak writes an audit_logs row (who, old→new, why) and can be reverted. A safe default always underlies the stack.

Policy vs preference

Org policy (director-only, affects all) is separated from personal preference (self-service). The UI makes the scope obvious.

Legible config

Show the current value, who set it, and when. No invisible behavior — a tweak you can't see is a bug you can't diagnose.

Show the evidence

A suggestion always carries its reasoning ("median accept 4m vs 10m window"), so the human can judge it, not just obey it.

Bounded & validated

Every tweak has a type and allowed range (Zod). Users can't push a timeout to 0 or a cap negative; guardrails stay server-side.

6 How it threads into the plan

Where it lands in the roadmap

This doesn't disturb the M0–M16 build. It slots in as additive milestones, and the cheapest tier could even come early since it extends org-config that already exists.

C1

Runtime settings layer extends existing org config

org_settings + user_preferences + a cached config service; Settings UI. ✓ change a tweak in-app, next action reflects it, no deploy.
C2

Behavior flags

feature_flags per org; gate one new behavior behind a flag. ✓ flip on one site without a release; instant rollback.
C3

Adaptive suggestions

suggestions + the analyze loop + a review card. ✓ system proposes a timeout/mapping change with evidence; human accepts → audited.
⚠️One thing to decide early: which tweaks are org policy vs user preference. Getting that boundary right up front avoids a painful reclassification later — and keeps clinical-routing behavior firmly on the director-controlled, audited side.
